# Agent Lifecycle

Agents in Loom OS follow a seven-stage lifecycle. All communication happens through the filesystem — agents drop files into `~/.loom/inbox/<project>/` and the daemon reacts.

## Lifecycle Stages

```
Registration → Graph Build → Heartbeat → Contribution → Dispatch → Re-registration
```

### 1. Registration

Agent writes `register.json` to `inbox/<project>/`:

```json
{
  "agent": "claude-code",
  "version": "2.1.190",
  "project": "noor",
  "project_path": "/abs/path/to/project",
  "capabilities": ["code-analysis", "refactoring"]
}
```

The daemon upserts the agent and project in the SQLite registry and queues a full Graphify build. Graph stats (node/edge counts) are preserved across re-registrations.

### 2. Graph Build

The daemon runs a full Graphify build on the project codebase. The knowledge graph is stored in `<project_path>/graphify-out/graph.json`. Stats are persisted to the registry. A `graph:updated` WebSocket event fires on completion.

### 3. Heartbeat

Agent writes `heartbeat.json` every ~60s:

```json
{
  "agent": "claude-code",
  "project": "noor",
  "status": "analyzing auth module",
  "timestamp": "2026-06-25T14:30:00Z"
}
```

Each heartbeat refreshes `last_heartbeat` and keeps the agent `online`. The daemon currently does not sweep for stale agents (no `agent:offline` event), but `last_heartbeat` is recorded for future use.

### 4. Contribution

Agent writes `finding-*.md` or `decision-*.md` as it discovers things:

- **Findings** (`finding-*.md`): Code analysis results. If `files` frontmatter references code files, an incremental graph update is queued.
- **Decisions** (`decision-*.md`): Architecture decision records (ADRs), processed as architecture-decision findings.

Each contribution triggers a `finding:ingested` WebSocket event.

### 5. Graph Update

The daemon ingests findings, queues incremental Graphify updates, and persists updated stats to the registry. Updates are rate-limited to one per project per 30 seconds (`MIN_UPDATE_INTERVAL`).

### 6. Dispatch

Work can be pushed to an agent via the dashboard or API. The daemon writes a `task-*.json` to the agent's inbox. An `agent:dispatched` WebSocket event fires.

For the full task execution flow (worker, worktrees, Kanban lifecycle), see [Task Board & Worker](TASK-BOARD.md).

### 7. Re-registration

A new `register.json` re-affirms online status. Graph stats are preserved. This allows an agent to rejoin a project after disconnecting without losing accumulated knowledge.

## Agent Status

| Status | Meaning |
|--------|---------|
| `online` | Heartbeat received within the last threshold |
| `offline` | No recent heartbeat (currently no sweep — stays at last status) |

## Capabilities

Agents declare capabilities during registration. These are free-form strings (e.g. `code-analysis`, `refactoring`, `testing`, `documentation`). The dashboard surface agent capabilities in agent cards and use them to filter dispatch targets.

## Shared Context

Every project has a `.loom/SHARED_CONTEXT.md` file auto-generated by the daemon. It contains graph stats, community lists, key symbols, knowledge sources, recent findings, and the agent roster. Agents can read this to orient themselves without API calls — just open the file.

## Supported Agents

Loom OS is agent-agnostic. Any process that can write files to `~/.loom/inbox/` can participate. The worker currently integrates with:
- **Claude Code** (`claude-code`)
- **OpenAI Codex** (`codex`)

Additional agents can be added by implementing the filesystem protocol.
